# ADR 0003: 日付の内部表現を ISO 8601 文字列に統一する

## Status

Accepted (2026-05-20)

## Context

履歴書 (履歴書) と職務経歴書 (職務経歴書) は日付フィールドを多く含みます。具体的には:

- 生年月日
- 学歴の入学・卒業日
- 職歴の入社・退社日
- 資格の取得日
- プロジェクトの期間

これらのフィールドは、日本式の文脈では和暦 (令和・平成・昭和等) で記載される慣習がある一方、実務では西暦表記も広く使われています。

内部データモデルとしては以下の選択肢があります。

1. **A. ISO 8601 文字列のみ (`"1993-04-01"` / `"2018-04"`)**
2. B. 西暦と和暦の両方を保存 (`{ gregorian: "1993-04-01", wareki: "平成5年4月1日" }`)
3. C. 和暦オブジェクトのみ (`{ era: 'heisei', year: 5, month: 4, day: 1 }`)

B は同じ情報を 2 つの形式で保存するため、入力経路によってデータの不整合が起きやすく、Single Source of Truth を保てません。C は西暦変換が必要、ソートが面倒、明治より前を扱えない等の制約があり、データモデルとして扱いにくくなります。

A はソート・比較・差分計算が容易で、外部システムとの相互運用性も高く、TypeScript の文字列リテラル型および brand 型と相性が良いです。和暦が必要なのは「表示時」だけであり、これは表示層 (renderer / template) の責務として後付け可能です。

## Decision

`packages/core` の日付フィールドは **ISO 8601 文字列のみ** で表現します。

- 日付 (年・月・日): `"YYYY-MM-DD"` 形式の `IsoDateString` ブランド型
- 年月のみ (職歴の開始/終了等): `"YYYY-MM"` 形式の `IsoYearMonthString` ブランド型

許可する年範囲は **1900〜2100** とします。履歴書・職務経歴書の用途として現実的な範囲であり、JavaScript の `Date` API における年 0〜99 の特殊な扱い (1900 年加算) を回避できます。

うるう年判定および月ごとの日数判定は手動で実装し、JavaScript の `Date` parser には依存しません。これは `2024-02-30` のような不正値を `Date` が丸めてしまうのを防ぐためです。

和暦表示が必要になった場合、それは **renderer / template の責務** として実装します。`packages/core` は和暦変換関数を持ちません。

## Consequences

**良い結果:**

- データソースが 1 つで明確 (ISO 8601 のみ)
- ソート・比較・差分計算が容易
- 外部システムとの相互運用性が高い
- 表示形式は表示層で自由に決められる (西暦 / 和暦 / 任意のフォーマット)

**負担となる結果:**

- 入力 UI で和暦が打鍵された場合、UI 側で西暦に変換してから core に渡す必要がある
- 和暦表示が必要になった時点で renderer / template に変換ロジックを実装する必要がある
- 1900 年より前、2100 年より後の日付は扱えない (歴史的経歴の記録には不向きだが、現代の履歴書用途では問題ない)

**これにより制約される後続判断:**

- `packages/core` に和暦変換 (`wareki()` / `fromWareki()` 等) を入れない
- 和暦が必要になった場合は、`feat/renderer-date-format` のような別 PR で renderer 側に実装する
